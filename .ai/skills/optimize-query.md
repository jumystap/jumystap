# Skill: optimize-query

Оптимизация запросов к БД в JUMYSTAP.

## Когда использовать
Медленные страницы/списки (объявления, вакансии, аналитика), подозрение на N+1, тяжёлые выборки.

## Последовательность
1. **Локализуй:** запросы живут в `app/Repositories/`. Найди метод репозитория, обслуживающий
   медленный экран (`AdRepository`, `AnnouncementRepository`, `AnalyticsRepository`, `HomeRepository`).
2. **Диагностика:** включи лог запросов (`DB::enableQueryLog()` локально) или Telescope (есть в
   зависимостях, `config/telescope.php`) и найди N+1/полные сканы.
3. **Исправь:**
   - eager loading (`with()`) вместо ленивой загрузки в циклах;
   - выбор только нужных колонок (`select`);
   - индексы — новой миграцией (см. database-agent), не правя старые;
   - пагинация для больших списков.
4. **Не меняй бизнес-результат** выборки — только её эффективность.

## Обязательные проверки
- Сравни кол-во/время запросов до и после.
- `./vendor/bin/pest` (поведение неизменно).
- Если добавлена миграция-индекс — `php artisan migrate`.

## Expected output
Меньше запросов/времени при идентичном результате; индексы добавлены миграцией; тесты зелёные.
Зафиксируй приём в `.ai/memory/architectural-decisions.md`, если он типовой для проекта.
