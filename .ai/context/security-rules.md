# Context: Security rules

## Секреты
- Никогда не коммить и не логировать: `APP_KEY`, БД-креды, `SMSC_*`, `BITRIX_URI`,
  `GENDER_API_KEY_*`, AWS-ключи. `.env` — вне образа (`.dockerignore`).
- В `.env.example` — только плейсхолдеры.

## Аутентификация и доступ
- Web — сессии (`database`, lifetime ~1 год — длинная сессия, учитывай при изменениях безопасности).
- Sanctum — токены для `/api/user`.
- Админка `/admin/*` — только admin/moderator через `AdminMiddleware`. Не обходить.
- `BlockedUserLogout` — заблокированные не должны иметь доступ.
- Авторские проверки контента — как `CheckAnnouncementAuthor`; не давай редактировать чужое.

## Валидация и ввод
- Весь пользовательский ввод — через FormRequest. Не доверять данным с фронта.
- Верификация по телефону (коды/SMSC) — учитывай троттлинг и перебор кодов (`VerificationController`,
  модель `Code`).

## Вывод
- HTML-контент пользователя на фронте санитизировать `dompurify`.
- Не выводить во фронт лишние поля пользователя (Inertia share отдаёт только id/name/email/role).

## Запросы и БД
- Использовать Eloquent/Query Builder (параметризация) — не конкатенировать сырой SQL с вводом.
- Файлы/изображения — через intervention/image + оптимизатор; валидировать типы/размеры загрузок.

## Внешние интеграции
- Bitrix/SMSC/Gender — таймауты и обработка ошибок; не падать всем приложением при недоступности.
- Telegram webhook (`POST /telegram/webhook`) — обрабатывать только ожидаемые команды/коллбеки
  (`Handler.php` уже отклоняет неизвестные).

## Чек перед мержем
- [ ] Нет секретов в диффе.
- [ ] Новые маршруты имеют корректный middleware (auth/admin/sanctum).
- [ ] Ввод валидируется FormRequest, вывод санитизируется.
- [ ] Нет сырого SQL с пользовательским вводом.
- [ ] Загрузки файлов валидируются.
